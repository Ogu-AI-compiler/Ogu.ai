# Process Improvement

## When to Use
- Engineering cycle time is longer than it should be
- A manual process is causing toil and errors
- Post-retrospective action items need to be turned into process changes

## Workflow
1. Measure the current state: cycle time, lead time, deployment frequency (DORA metrics)
2. Map the value stream: identify the biggest bottlenecks and waste categories
3. Prioritize improvements by impact vs effort; start with quick wins
4. Implement the change and define the metric that proves it worked
5. Review the change after 4-6 weeks; keep, adjust, or revert based on data

## Quality Bar
- Process changes are based on measured baselines, not anecdotes
- Every change has a defined success metric and review timeline
- The team participates in designing process changes — imposed processes don't stick
- Improvements are iterated incrementally, not in big-bang rewrites
